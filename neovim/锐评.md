我会给这份配置：

```text
8.2 / 10
```

```text
Neovim 0.12+ 原生能力
+ lazy.nvim
+ blink.cmp
+ vim.lsp.config()/vim.lsp.enable()
```

这条线是对的。它没有走 LazyVim/AstroNvim 那种全家桶路线，也没有把补全、AI、LSP、formatter 搅成一锅粥。整体是“自己维护一份现代轻量配置”的路子。

但新版 `init.lua` 里最需要重新评价的是 Tree-sitter：

```lua
{
  "romus204/tree-sitter-manager.nvim",
  lazy = false,
  priority = 1000,
  config = function()
    require("tree-sitter-manager").setup({
      parser_dir = vim.fn.stdpath("data") .. "/site/parser",
      query_dir = vim.fn.stdpath("data") .. "/site/queries",

      ensure_installed = {
        "rust",
        "python",
        "cpp",
      },

      auto_install = false,

      highlight = {
        "rust",
        "python",
        "cpp",
      },

      noauto_install = {
        "c",
        "lua",
        "markdown",
        "markdown_inline",
        "query",
        "vim",
        "vimdoc",
      },

      nerdfont = true,
    })
  end,
}
```

这段已经意识到 Neovim 0.12 自带部分 parser，所以把 `c/lua/markdown/vim/vimdoc` 放进了 `noauto_install`。这个意识是对的。但现在的写法还是有点“脚踩两条船”：

* `auto_install = false` 时，`noauto_install` 基本没有发挥核心价值。
* `highlight = { "rust", "python", "cpp" }` 只让 manager 管这三个语言的高亮，内置 parser 对应的 C/Lua/Markdown/Vim/Vimdoc 是否启动高亮，要看你有没有别的 `vim.treesitter.start()` 入口。
* 如果你已经目标锁定 Neovim 0.12，那么 Tree-sitter 段落应该从“安装很多 parser”改成“内置的交给 Nvim，缺的语言才交给 manager”。

所以这版的主要短板不是“插件不够”，而是：**0.12 原生 Tree-sitter 和第三方 parser manager 的边界还没切干净。**

## 1. 版本目标应该明确写成 Neovim 0.12+

上一版还在说 0.11，这版建议直接改口：这份配置就是 Neovim 0.12+ 配置。

原因有三个：

第一，blink.cmp 当前文档要求 Neovim 0.12+。你用了：

```lua
{
  "saghen/blink.cmp",
  version = "1.*",
  ...
}
```

这已经天然把版本基线推到了 0.12。

第二，你已经用的是现代 LSP API：

```lua
vim.lsp.config("clangd", { ... })
vim.lsp.config("pyright", { ... })
vim.lsp.config("ruff", { ... })

vim.lsp.enable({
  "clangd",
  "pyright",
  "ruff",
})
```

`nvim-lspconfig` 当前 README 也明确说旧的 `require('lspconfig')` framework 已 deprecated，推荐 0.11+ 用 `vim.lsp.config()` 和 `vim.lsp.enable()`。

第三，Tree-sitter-manager.nvim 自己也把目标写成 Neovim 0.12+，因为它的定位就是：在 0.12 已集成 Tree-sitter core、但仍缺 parser installer 的情况下，做一个轻量 parser 管理器。

建议在配置最前面加版本保护：

```lua
if vim.fn.has("nvim-0.12") == 0 then
  error("This config requires Neovim 0.12+")
end
```

这比继续写 `nvim-0.11` 更准确。你现在不是“兼容老版本”的配置，而是“吃 0.12 原生能力红利”的配置。

## 2. Tree-sitter 段落要重写：0.12 内置 parser 不该再重复管理

Neovim 当前 Tree-sitter 文档写得很清楚：Nvim 集成了 tree-sitter library，并且自带这些 parser：

```text
C
Lua
Markdown
Vimscript
Vimdoc
```

此外，Nvim 会在 runtimepath 里找 `parser/{lang}.*`，如果同一个语言有多个 parser，优先级通常是：

```text
用户配置 > 插件 > 内置
```

这意味着一个很现实的问题：你如果额外装了 C/Lua/Markdown/Vim/Vimdoc parser，反而可能让外部 parser 覆盖 Nvim 自带 parser。不是一定坏，但维护面会变大。

所以你的 `noauto_install` 思路是对的：

```lua
noauto_install = {
  "c",
  "lua",
  "markdown",
  "markdown_inline",
  "query",
  "vim",
  "vimdoc",
},
```

但现在的问题是你又写了：

```lua
auto_install = false,
```

这导致 `noauto_install` 的实际意义被削弱。`noauto_install` 最适合和 `auto_install = true` 搭配：遇到新 filetype 时自动装 parser，但跳过 Nvim 已经内置、或者你不想自动装的语言。

### 建议写法 A：保留 tree-sitter-manager，让它只负责非内置 parser

这是我最推荐的方案：

```lua
{
  "romus204/tree-sitter-manager.nvim",
  lazy = false,
  priority = 1000,
  config = function()
    require("tree-sitter-manager").setup({
      parser_dir = vim.fn.stdpath("data") .. "/site/parser",
      query_dir = vim.fn.stdpath("data") .. "/site/queries",

      ensure_installed = {
        "rust",
        "python",
        "cpp",
      },

      -- 让缺失语言按需安装，但跳过 Nvim 0.12 已内置/不想自动装的语言。
      auto_install = true,
      noauto_install = {
        "c",
        "lua",
        "markdown",
        "markdown_inline",
        "query",
        "vim",
        "vimdoc",
      },

      -- 让 manager 负责启动它知道的 parser；内置 parser 仍可由 Nvim start。
      highlight = true,

      nerdfont = true,
    })
  end,
}
```

这段的语义更干净：

```text
Nvim 0.12 自带的：别装。
项目真正需要但 Nvim 不带的：让 manager 装。
高亮启动：由 manager + 原生 vim.treesitter.start() 配合。
```

### 建议写法 B：内置语言显式用 `vim.treesitter.start()`

不管你保不保留 manager，我都建议给内置 parser 一个清晰入口：

```lua
vim.api.nvim_create_autocmd("FileType", {
  pattern = {
    "c",
    "lua",
    "markdown",
    "vim",
    "vimdoc",
  },
  callback = function(event)
    pcall(vim.treesitter.start, event.buf)
  end,
})
```

如果你想一并开启 Tree-sitter fold：

```lua
vim.api.nvim_create_autocmd("FileType", {
  pattern = {
    "c",
    "lua",
    "markdown",
    "vim",
    "vimdoc",
  },
  callback = function(event)
    pcall(vim.treesitter.start, event.buf)
    vim.wo.foldmethod = "expr"
    vim.wo.foldexpr = "v:lua.vim.treesitter.foldexpr()"
  end,
})
```

不过你现在已经有：

```lua
vim.opt.foldlevel = 99
vim.opt.foldlevelstart = 99
```

所以 fold 这块可以晚点再动。优先保证 highlight 启动就行。

### 对这段配置的锐评

这段不是错，而是“知道 0.12 有内置 parser，但还没完全按 0.12 的方式组织”。

上一版我会说“Lua 没装 parser”。现在不能这么说，因为 Neovim 0.12 本来就带 Lua parser。真正的问题变成：**你没有在配置层面把‘内置 parser 如何启动’写清楚。**

## 3. LSP 方向正确，但现在可以更 0.12 化

你这段仍然是优点：

```lua
local capabilities = require("blink.cmp").get_lsp_capabilities()

vim.lsp.config("clangd", {
  capabilities = capabilities,
  cmd = {
    "clangd",
    "--background-index",
    "--clang-tidy",
    "--completion-style=detailed",
    "--header-insertion=iwyu",
    "--compile-commands-dir=build",
  },
  root_markers = {
    "compile_commands.json",
    "compile_flags.txt",
    ".clangd",
    ".git",
  },
})
```

`vim.lsp.config()` / `vim.lsp.enable()` 是正确方向。这个不需要退回 `require("lspconfig").clangd.setup({})`。

但有两个细节。

### clangd 的 `--compile-commands-dir=build` 仍然有假设

这段只适合你所有 C/C++ 项目都长这样：

```text
project/
  build/
    compile_commands.json
  src/
```

如果你经常有：

```text
build-debug/
build-release/
cmake-build-debug/
out/build/
```

那 clangd 就会找错或者找不到 compilation database。

如果你 workflow 固定是 `build/`，保留没问题。如果不是，我仍然建议把 `compile_commands.json` 放到项目根目录或软链接到根目录，然后删掉 `--compile-commands-dir=build`。

### ruff 建议关 hover

现在：

```lua
vim.lsp.config("ruff", {
  capabilities = capabilities,
})
```

建议改成：

```lua
vim.lsp.config("ruff", {
  capabilities = capabilities,
  on_attach = function(client)
    client.server_capabilities.hoverProvider = false
  end,
})
```

这样 `K` 的 hover 主要由 pyright 负责，ruff 继续做 lint/code action。否则 Python buffer 里多个 LSP 抢 hover，手感会有点乱。

## 4. Copilot 配置总体正确，但 filetypes 仍然太保守

你现在：

```lua
filetypes = {
  python = true,
  cpp = true,
  ["*"] = false,
},
```

这句话的含义非常强：**只允许 Python 和 C++，其他全部禁用。**

如果这是刻意的，很好，隐私边界很清楚。但从你的配置看，你还在写 Lua 配置、装了 Rust parser、也有 C formatter：

```lua
formatters_by_ft = {
  c = { "clang_format" },
  cpp = { "clang_format" },

```

所以更自然的写法是：

```lua
filetypes = {
  python = true,
  cpp = true,
  c = true,
  rust = true,
  lua = true,
  markdown = true,
  gitcommit = true,
  ["*"] = false,
},
```

或者反过来，只禁用你敏感的 filetype，让 Copilot 使用默认策略：

```lua
filetypes = {
  gitcommit = true,
  markdown = true,
  yaml = false,
  env = false,
}
```

不过我更偏向你现在这种白名单思路，只是白名单应该至少覆盖你真的在写的语言。

另外，`copilot.lua` 现在要求 Neovim 0.11+，如果使用默认 nodejs LSP，还要求 Node.js v22+。所以这条建议保留：

```bash
node --version
```

如果不是 v22+，就给 `copilot_node_command` 指到明确的 Node 22 路径。

NES 这里：

```lua
require("copilot-lsp").setup({
  nes = {
    move_count_threshold = 10,
  },
})
```

我仍然觉得 10 偏大。建议：


或者折中：

```lua
move_count_threshold = 5,
```

NES 本来就是实验性功能，过期 suggestion 留太久会让 Normal mode 的动作变得有点“鬼影追踪”。

## 5. `<leader>r` 很实用，但应该加 filetype guard

你现在有一个挺顺手的 Python runner：

```lua
vim.keymap.set("n", "<leader>r", function()
  local file = vim.fn.expand("%:p")
  local root = vim.fs.root(0, { "pyproject.toml", ".git" }) or vim.fn.getcwd()

  vim.cmd("botright 12split")
  vim.cmd(
    "terminal cd "
      .. vim.fn.shellescape(root)
      .. " && uv run python "
      .. vim.fn.shellescape(file)
  )
  vim.cmd("startinsert")
end, { desc = "Run current Python file with uv" })
```

这个比很多人配的 `:!python %` 强，至少知道找项目 root，也知道用 `uv run`。

但它现在对任何 filetype 都可触发。建议加 guard：

```lua
vim.keymap.set("n", "<leader>r", function()
  if vim.bo.filetype ~= "python" then
    vim.notify("<leader>r is only configured for Python files", vim.log.levels.WARN)
    return
  end

  local file = vim.fn.expand("%:p")
  local root = vim.fs.root(0, { "pyproject.toml", ".git" }) or vim.fn.getcwd()

  vim.cmd("botright 12split")
  vim.cmd(
    "terminal cd "
      .. vim.fn.shellescape(root)
      .. " && uv run python "
      .. vim.fn.shellescape(file)
  )
  vim.cmd("startinsert")
end, { desc = "Run current Python file with uv" })
```

否则你在 Lua/C++ buffer 里按错，开个 terminal 去跑 `uv run python some.lua`，就挺荒谬。

```lua
        gs.reset_hunk({ vim.fn.line("."), vim.fn.line("v") })
      end, "Reset selected hunk")

      map("n", "<leader>hS", gs.stage_buffer, "Stage buffer")
      map("n", "<leader>hR", gs.reset_buffer, "Reset buffer")

      map("n", "<leader>hb", function()
        gs.blame_line({ full = true })
      end, "Blame line")

      map("n", "<leader>hd", gs.diffthis, "Diff this")
      map("n", "<leader>hD", function()
```

```lua
  markdown = true,
  gitcommit = true,
  ["*"] = false,
},
```

### 第五，ruff 关 hover

```lua
vim.lsp.config("ruff", {
  capabilities = capabilities,
  on_attach = function(client)
    client.server_capabilities.hoverProvider = false
  end,
})
```

## 最终判断

这份配置现在的问题不是“落后”，而是“有些地方已经进入 0.12 时代，有些地方还残留 0.11 / nvim-treesitter 时代的心智模型”。

最典型的就是 Tree-sitter：

```text
旧思路：列 ensure_installed，装齐 parser，插件负责一切。
新思路：Nvim 0.12 已内置的别装；缺的 parser 再用 manager；高亮启动用 vim.treesitter.start() 明确表达。
```

核心架构我不建议大改：

```text
lazy.nvim                       插件管理
rose-pine dawn                  主题
blink.cmp                       补全菜单
copilot.lua                     inline AI suggestion
copilot-lsp                     NES
nvim-lspconfig                  LSP server config source
conform.nvim                    formatter
native vim.treesitter           0.12 内置 Tree-sitter 能力
tree-sitter-manager.nvim        非内置 parser 管理
gitsigns.nvim                   buffer 内 Git hunk
vim-fugitive                    Git command workflow
oil.nvim                        文件管理
nvim-dap-python                 Python 调试
```

我会把下一步优先级排成这样：

```text
1. Tree-sitter 按 0.12 思路重整
2. Copilot + blink 双向隐藏
3. Git/DAP 键位补齐
4. ruff hover 关闭
5. Copilot filetypes 扩大
6. <leader>r 加 Python filetype guard
7. 有空再做 lazy-load 精修
```

一句话锐评：

> 这份配置已经不是“新手拼插件”了，是一份有明确审美的轻量 Neovim 0.12 配置；但 Tree-sitter 那段还停在“知道内置 parser 存在”而不是“真正按内置 parser 组织配置”。先把这段理顺，整份配置会干净很多。
